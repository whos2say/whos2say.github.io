# DJR Participant CMS Status

This document describes the current DJR Photography participant page and the smallest safe path toward a participant-facing CMS model.

## Current architecture

The DJR page is a static participant microsite under `/djr/`.

- `djr/index.html` is the public landing page for DJR Photography.
- `djr/galleries.html` and `djr/contact.html` are supporting static pages.
- `djr/js/djr-content.js` loads JSON from `content/djr` and renders page copy, chrome, contact text, and gallery page text.
- `djr/js/djr-section-galleries.js` loads `content/djr/section-gallery-map.json` and renders gallery cards into DJR homepage sections.
- `content/djr/home.json` stores the main DJR homepage content.
- `content/djr/site.json` stores DJR-local branding, nav, contact, footer, social, and partner data.
- `content/djr/galleries.json` stores gallery page labels, categories, messages, and optional curated gallery entries.
- `content/djr/contact.json` stores contact page copy plus the form action and session types.
- `content/djr/section-gallery-map.json` maps homepage sections to CMS albums, Photo Gallery App albums, or Google Photos URLs.
- `content/djr-albums/*.json` stores Decap-managed DJR CMS album content.

The shared Decap source is `admin/config.shared.yml`. `admin/config.yml` is generated by `scripts/generate-decap-config.mjs`, so CMS schema changes should be made in the shared file.

## Regular CMS vs participant CMS

The regular CMS is for Who's to Say staff or trusted admins. It can edit global website content such as the homepage, navigation, site settings, program pages, stories, public classes, and shared configuration.

The participant CMS should be a narrower editing surface for one participant area, starting with `/djr/`. It should help a participant or support person update their voice, images, albums, captions, and simple public-facing copy without gaining access to global site structure or other participants.

The participant CMS should not expose:

- Global navigation or site settings.
- Layout structure or arbitrary HTML.
- Other participant content.
- Formspree or form action URLs.
- Photo Gallery App IDs.
- Global Who's to Say branding, footer, or partner settings.
- Production-sensitive routing or deployment settings.

## Current CMS exposure

Decap currently exposes the `djr-gallery-albums` collection:

- Source: `admin/config.shared.yml`
- Folder: `content/djr-albums`
- Create new album files: yes
- Safe fields: `album_id`, `title`, `description`, `cover_image`, `images`, `image`, `alt`, `caption`, `credit`, `tags`, `orientation`

This is a good participant CMS starting point because it is scoped to DJR album content and does not grant access to global site files.

Decap also exposes the `djr-participant-copy` file collection:

- Source: `admin/config.shared.yml`
- File: `content/djr/participant-copy.json`
- Purpose: participant-editable homepage copy overlay for the DJR landing page
- Safe fields: hero tagline; story copy; about copy and portrait; creative feature copy and images

This collection does not expose DJR nav, footer, form actions, route links, Photo Gallery App IDs, Google Photos URLs, or global Who's to Say content.

## Participant-editable fields

These fields are reasonable for a participant-facing editor once they are represented in a safe schema:

- Album title and description.
- Album cover image.
- Album images.
- Image alt text.
- Image captions.
- Image credit.
- Image tags.
- Image orientation.
- Homepage hero tagline.
- Story section eyebrow, title, lead, body, and quote.
- About section eyebrow, title, body, and portrait.
- Featured story title, body, label, and image when not tied to a Photo Gallery App ID.
- Creative feature eyebrow, title, body, images, and alt text.
- Service card labels, short descriptions, icons from a fixed list, images, and categories.
- Simple section visibility toggles where the section itself is fixed by the template.

## Admin-only fields

These should remain staff/admin-owned for now:

- `content/site.json` and `content/navigation.json`.
- Global homepage, program, story, class, and support coordinator content.
- DJR-local nav in `content/djr/site.json`.
- DJR footer, partner logo/link, and copyright.
- DJR contact form action in `content/djr/contact.json`.
- Photo Gallery App album IDs and direct `/album.html?album=...` URLs.
- Google Photos shared album URLs unless staff approves the public sharing model.
- Button hrefs and route-level links.
- Section IDs and mapping source types in `content/djr/section-gallery-map.json`.
- Any unrestricted HTML fields.

## Automated DJR content test

Run:

```bash
npm run test:djr-content
```

The test is dependency-free and runs with Node. It currently checks:

- `/djr/` exists and loads the DJR renderers.
- DJR identity and required homepage content exist.
- Local image references in `content/djr` and `content/djr-albums` resolve.
- At least one DJR CMS album fixture exists.
- CMS album IDs are slug-safe.
- Album images include image paths and alt text.
- Enabled CMS album mappings point to real album files.
- Mapped CMS albums contain at least one image for public rendering.
- The section gallery renderer includes safe missing-album and empty-album fallback messages.
- The current DJR CMS album collection is scoped to `content/djr-albums` and does not expose global/admin paths.

This is not a full browser test. If the repo later adds Playwright, the next browser smoke test should load `/djr/`, wait for DJR content render, and assert that the story/about/gallery areas appear without console errors.

## Smallest safe Decap expansion

Do not expose `content/djr/home.json` directly. It mixes safe participant copy with admin-owned URLs, fixed routes, images, service card hrefs, and Photo Gallery App album links. A direct Decap file collection would make it too easy to mutate routing or album wiring.

The implemented smallest safe expansion is:

1. `content/djr/participant-copy.json` contains only participant-editable copy and image fields.
2. `djr/js/djr-content.js` loads `participant-copy.json` after `home.json` and overlays only known safe fields.
3. `admin/config.shared.yml` includes a Decap file collection for `content/djr/participant-copy.json`.
4. `npm run test:djr-content` proves the participant copy file and collection do not include forbidden paths or fields such as `href`, `formAction`, `photoGalleryAlbumId`, `googlePhotosAlbumUrl`, `nav`, or `partner`.

Suggested first safe fields for that collection:

- `hero.tagline`
- `story.enabled`
- `story.eyebrow`
- `story.title`
- `story.lead`
- `story.body`
- `story.quote`
- `about.enabled`
- `about.eyebrow`
- `about.title`
- `about.body`
- `about.photo`
- `creativeFeature.enabled`
- `creativeFeature.eyebrow`
- `creativeFeature.title`
- `creativeFeature.body`
- `creativeFeature.images[].src`
- `creativeFeature.images[].alt`

Leave buttons, hrefs, route links, Photo Gallery App IDs, Google Photos URLs, nav, contact form configuration, partner/footer data, and section mapping source types admin-only.

## Sprint 1 recommendation

Keep `djr-gallery-albums` and `djr-participant-copy` as the only participant-editable Decap surfaces until there is a second participant template or a stronger role/scope model.

That sequencing avoids over-refactoring and keeps the public page behavior stable while creating a repeatable participant CMS pattern.
